## WI-BOT-002 — Telegram webhook ingress + secret validation

- Story ID: WI-BOT-002
- MoSCoW: Must
- Links (RS/D): RS-A-01, RS-A-07
- UAT: BOT-UAT-09

Acceptance Criteria:
- Webhook endpoint принимает входящие Telegram события и не падает при некорректных/неполных payload (ошибки обрабатываются контролируемо).
- Если `TELEGRAM_WEBHOOK_SECRET_TOKEN` задан, запросы без корректного секрета отклоняются.
- Для каждого входящего события создается audit/log запись, достаточная для подтверждения BOT-UAT-09 (тип, время, chat/user, связка с обработкой).
- Дубликаты Telegram update не приводят к повторной бизнес-обработке (idempotency на уровне update_id/эквивалента).
- Связь “входящий update → диалог/эскалация/смета” сохраняется и доступна для проверки.
- Наличие/отсутствие manager notifications (chat id) не влияет на устойчивость приема webhook.

## WI-BOT-011 — Первичная квалификация (сбор 15 параметров)

- Story ID: WI-BOT-011
- MoSCoW: Must
- Links (RS/D): RS-A-01
- UAT: BOT-UAT-01

Acceptance Criteria:
- Бот собирает 15 параметров клиента согласно baseline списка (см. `pre_kp_customer_questionnaire_and_input_data.md`) и умеет уточнять недостающие.
- При неполном вводе бот задает уточняющие вопросы и не завершает квалификацию “пустым” результатом.
- Прогресс квалификации сохраняется (контекст диалога), корректно восстанавливается при продолжении.
- По завершении квалификации формируется структурированная “карточка запроса” (для лида/менеджера).
- История сообщений сохраняется и доступна для просмотра менеджером (через хранилище/экспорт).
- BOT-UAT-01 проходит без ручных вмешательств со стороны менеджера.

## WI-BOT-012 — Нестандартный кейс → эскалация

- Story ID: WI-BOT-012
- MoSCoW: Must
- Links (RS/D): RS-A-01, RS-A-05, D-007
- UAT: BOT-UAT-04, BOT-UAT-07

Acceptance Criteria:
- Для сценариев вне подтвержденных правил/источников бот не выдает “финальный” ответ, а инициирует эскалацию менеджеру.
- Эскалация содержит причину + весь релевантный контекст (собранные параметры, история, вопрос пользователя).
- Для “неизвестного вопроса” бот гарантированно не придумывает ответ (BOT-UAT-07).
- Для “нестандартного расчета/условий” бот не применяет калькулятор v1 и эскалирует с причиной (BOT-UAT-04).
- Эскалации фиксируются в журнале/аудите как отдельные события (для BOT-UAT-09).
- Повторная эскалация (после уточнений пользователя) не теряет ранее собранный контекст.

## WI-BOT-013 — Правило takeover менеджером + `/resume_bot`

- Story ID: WI-BOT-013
- MoSCoW: Must
- Links (RS/D): RS-A-01
- UAT: BOT-UAT-08

Acceptance Criteria:
- После исходящего сообщения менеджера в чате автоматические ответы бота прекращаются (BOT-UAT-08).
- Возврат бота возможен только по явной команде `/resume_bot` или через админку (в соответствии с baseline).
- Состояния paused/resumed фиксируются в журнале/аудите с timestamp и инициатором.
- В состоянии paused бот не перехватывает диалог и не мешает менеджеру вести переписку.
- При смене статуса корректно обрабатываются гонки (приоритет менеджера над ботом).
- Поведение документировано как правило MVP и воспроизводимо в UAT.

## WI-BOT-014 — Бюджетная валидация `< 1 500 000` и корректный отказ

- Story ID: WI-BOT-014
- MoSCoW: Must
- Links (RS/D): RS-A-01
- UAT: BOT-UAT-02, CALC-UAT-02

Acceptance Criteria:
- При вводе бюджета ниже 1 500 000 RUB бот корректно сообщает об ограничении и не обещает “целевой подбор” (BOT-UAT-02).
- Бот предлагает пользователю понятные варианты действий: уточнить параметры/альтернативы/передача менеджеру.
- В этом сценарии бот не выдает вводящую в заблуждение финальную цену и не “подменяет” решение калькулятором.
- Событие “budget below threshold” фиксируется в контексте/лиде и журнале.
- Пользователь может продолжить диалог (уточнения/параметры) без сброса контекста.
- CALC-UAT-02 соблюдается как бизнес-правило, не нарушая другие BOT-UAT сценарии.

## WI-BOT-015 — Ответы по базе знаний: source-gating + “unknown → escalate”

- Story ID: WI-BOT-015
- MoSCoW: Must
- Links (RS/D): RS-A-02, RS-A-05, D-007, `knowledge_base_corpus_register.md`
- UAT: BOT-UAT-05, BOT-UAT-07

Acceptance Criteria:
- Бот использует для финальных ответов только подтвержденные источники (`approved`) из реестра KB.
- Если релевантного подтвержденного источника нет или уверенность низкая — бот эскалирует менеджеру (BOT-UAT-07).
- Ответ по комплектации/характеристикам содержит ссылку на источник или явную оговорку об ограничениях/уровне уверенности (BOT-UAT-05).
- При конфликте источников бот не выбирает “самый правдоподобный” вариант, а эскалирует.
- Факт использования источника фиксируется (какой источник/когда) для проверки качества и повторяемости.
- MVP соблюдает правило: запрет на автоматический парсинг/копирование контента сторонних сайтов; допускаются ссылки и справочные метаданные (см. KB policy).

## WI-BOT-016 — Shortlist 3–5 альтернатив (в рамках данных) или эскалация

- Story ID: WI-BOT-016
- MoSCoW: Must
- Links (RS/D): RS-A-01, RS-A-02
- UAT: BOT-UAT-06

Acceptance Criteria:
- При достаточных данных бот формирует shortlist из 3–5 альтернатив (модели/варианты) и кратко объясняет, почему они подходят (BOT-UAT-06).
- Если данных недостаточно, бот либо задает уточняющие вопросы, либо эскалирует менеджеру; “вымышленных” вариантов быть не должно.
- Если выдача альтернатив опирается на KB, используются только подтвержденные источники; иначе — эскалация.
- Shortlist фиксируется в журнале/контексте с входными параметрами (для контроля качества).
- Пользователь может скорректировать параметры и получить обновленный shortlist без потери контекста.
- Shortlist не противоречит ограничениям бюджета/контура калькулятора/правилам эскалации.

## WI-CALC-010 — Расчет и смета с версией/курсами/timestamp

- Story ID: WI-CALC-010
- MoSCoW: Must
- Links (RS/D): RS-A-03, D-003, D-004, `calculator_formula_specification.md`
- UAT: CALC-UAT-01, BOT-UAT-03

Acceptance Criteria:
- Расчет выполняется только при выполнении условий контура v1 (РФ, 3–5 лет, ≤160 л.с., подтвержденные ставки/курсы).
- В результате формируется смета с разбивкой по статьям по baseline v1.
- Смета содержит: timestamp, версию формулы, источник курса и использованный курс.
- Изменение курса/ставки не перезаписывает старую смету: создается новый расчет (история сохраняется).
- При отсутствии подтвержденного курса или ставки бот не выдает финальную цену и инициирует эскалацию/ручную проверку.
- BOT-UAT-03 и CALC-UAT-01 проходят без “скрытых допущений” (все числа объяснимы и трассируемы).

## WI-CALC-020 — Валидация контура v1 и причины эскалации

- Story ID: WI-CALC-020
- MoSCoW: Must
- Links (RS/D): D-003, `calculator_formula_specification.md`
- UAT: CALC-UAT-03, CALC-UAT-04, CALC-UAT-05, CALC-UAT-06

Acceptance Criteria:
- До расчета происходит явная проверка контура v1 по стране/возрасту/мощности/типу авто.
- Для РБ/КЗ и иных стран расчет не применяется; формируется эскалация с указанием причины (CALC-UAT-05).

- Для мощности >160 л.с. и для электромобилей формируется эскалация с причиной (CALC-UAT-04).
- При отсутствии подтвержденного курса валюты бот не выдает финальную цену и фиксирует необходимость ручной проверки (CALC-UAT-06).
- Причины отказа/эскалации фиксируются как структурированное поле (для отчетности и улучшений).
