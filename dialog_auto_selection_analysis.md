# Анализ диалогов клиента по подбору авто и дополнение требований

> Дата анализа: 13.05.2026
> Источник: Диалог 1 и Диалог 2 (переписка менеджер ↔ клиент)
> Цель: дополнить входные данные для проектирования, оценки и разработки продукта.

## 1) Что уже подтверждено из реального поведения пользователей

- Пользователь ведёт **многошаговый выбор** между несколькими моделями и классами (седан/лифтбек/кроссовер), часто меняет фокус в одном диалоге.
- Ключевой вопрос — **итоговая цена «под ключ»** с раздельной детализацией: стоимость авто, расходы по стране покупки, комиссия, брокер, таможня, логистика до города клиента.
- Пользователь регулярно сравнивает **страны и каналы поставки** (Китай/Корея), а также последствия по утилю, типу ДВС, пробегу и состоянию.
- Есть высокий спрос на **технические уточнения по комплектациям**: люк/панорама, размер дисков, число комплектаций, признаки максимальной версии.
- Пользователь просит **альтернативы в заданном бюджете** и ожидает консультативную роль, а не только выдачу прайс-листа.
- География влияет на стоимость: в диалогах повторяется расчёт до Владивостока и отдельно до Воронежа (автовоз).
- Критично доверие к данным: клиент сомневается в точности внешних площадок («на дроме не точно пишут») и просит верификацию.
- Важны нефинансовые факторы: ликвидность/класс, «не хуже текущей машины», русификация интерфейса, доступность CarPlay, качество сборки/антикор.

## 2) Выявленные пользовательские интенты и сценарии (для CJM / use cases)

## UC-01: Первичный ценовой запрос
**Триггер:** «Сколько стоит в мой город?»
**Ожидание:** быстрый ответ с разбивкой по конфигурациям и логистике.
**Нужно в продукте:**
- автоопределение города клиента из текста;
- расчёт минимум в двух режимах: «до Владивостока» и «до города клиента»;
- хранение версии курса и времени расчёта.

## UC-02: Сравнение моделей в рамках бюджета
**Триггер:** «Бюджет около 2 млн, что посоветуете?»
**Ожидание:** короткий shortlist с плюсами/минусами и рисками.
**Нужно в продукте:**
- recommendation-правила по сегментам (седан/лифтбек/кроссовер), размеру, бренду;
- объяснимые рекомендации (почему эта модель попала в подборку);
- подсказка «что добавится при +X бюджета».

## UC-03: Проверка состава итоговой цены
**Триггер:** «Из чего складывается итог?»
**Ожидание:** прозрачная смета.
**Нужно в продукте:**
- компонентная калькуляция с формулами;
- шаблон вывода: сумма + комментарий по каждому компоненту;
- логирование версий тарифов (брокер, комиссия, логистика, таможня).

## UC-04: Техническая верификация комплектации
**Триггер:** «Максималка? Панорама? Какие колёса?»
**Ожидание:** точный ответ с источником и оговорками.
**Нужно в продукте:**
- база комплектаций с вероятностными признаками по рынкам;
- статус уверенности ответа (высокая/средняя/низкая);
- флаг «требуется ручная проверка менеджером».

## UC-05: Уточнение эксплуатационных ожиданий
**Триггер:** «Хочу не хуже текущей машины»
**Ожидание:** фильтрация по уровню комфорта/динамики/класса.
**Нужно в продукте:**
- профиль текущего авто клиента;
- правила сопоставления «не хуже чем» по классу и опциям;
- предупреждения о компромиссах.

## 3) Функциональные требования, которые необходимо добавить

1. **Модуль «Коммерческий калькулятор»**
   - Расчёт полной стоимости по формуле: цена авто + расходы по стране + комиссия + брокер + таможня + логистика.
   - Поддержка сценариев: фиксированные ставки, диапазоны, условные надбавки.
   - Сохранение расчётов в карточке лида (история пересчётов).

2. **Модуль «Конфигуратор комплектаций»**
   - Единый каталог комплектаций по моделям и рынкам (Китай/Корея).
   - Поля: двигатель, коробка, привод, опции (люк/панорама), диски, год, пробег, признаки «макс/предмакс».
   - Указание источника и уровня достоверности.

3. **Модуль «Рекомендатор альтернатив»**
   - Подбор похожих моделей при ограниченном бюджете.
   - Пояснение trade-off: размер/класс/мощность/стоимость владения.
   - Отдельные сценарии для клиента «хочу выше посадку».

4. **Модуль «Диалог и эскалация»**
   - Автоэскалация менеджеру при вопросах с низкой уверенностью или спорными данными площадок.
   - Передача контекста: что спросил клиент, какие варианты обсуждались, какой бюджет назвал.

5. **Модуль «Контроль качества ответов»**
   - Проверка единиц измерения, валюты, даты курса.
   - Анти-противоречия по комплектациям (если в базе 2 взаимоисключающих признака).

## 4) Данные, которые нужно начать собирать уже на этапе MVP

- Город клиента и целевой город доставки.
- Бюджет (целевой и предельный).
- Текущий автомобиль клиента и опыт владения.
- Приоритет кузова/класса, брендовые предпочтения.
- Минимальные требования к опциям (люк/панорама, мультимедиа, язык интерфейса).
- Допустимость: пробег, окрасы, год.
- Канал поставки и рынок интереса (Китай/Корея).
- Допустимый срок поставки.

## 5) Нефункциональные требования из контекста диалогов

- **Объяснимость:** каждый ценовой ответ должен быть декомпозирован.
- **Трассируемость:** хранить источник данных и дату актуальности.
- **Скорость:** первая смета — в течение одного пользовательского цикла общения.
- **Консистентность:** одинаковые ответы при повторном вопросе в рамках одной сессии.
- **Аудит:** фиксировать, где ответ дал бот, а где менеджер.

## 6) Противоречия и риски, обнаруженные в переписке

- «Максималка» и наличие панорамы/люка трактуются по-разному в объявлениях и фактических авто.
- Клиент сравнивает цены между сообщениями в разное время, но редко видит объяснение влияния курса.
- Термины («лонговый пассат», «предмаксимальная») не стандартизированы в публичных источниках.

**Что сделать в продукте:**
- словарь нормализации терминов;
- обязательный timestamp расчёта;
- карточка «что подтверждено / что требует проверки».

## 7) Дополнение к оценке проекта (scope MVP)

### Включить в MVP обязательно
- Скриптуемая анкета потребностей клиента (бюджет, класс, приоритеты).
- Калькулятор «под ключ» с разбивкой по статьям.
- Shortlist из 3–5 альтернатив с объяснением выбора.
- Эскалация менеджеру с передачей истории диалога.

### Отложить на Phase 2
- Автоматическое распознавание объявлений по ссылкам и автопарсинг характеристик.
- Предиктивная оценка ликвидности модели.
- Персональные рекомендации на основе исторических конверсий.

## 8) Вопросы заказчику для закрытия пробелов перед архитектурой

1. Какие именно формулы и ставки являются нормативными для расчёта цены «под ключ»?
2. Какие источники считаются эталонными по комплектациям (официальные каталоги, дилерские выгрузки, внутренние таблицы)?
3. Нужно ли фиксировать курс валюты на момент первого расчёта и «замораживать» оффер на срок?
4. Какой SLA ответа менеджера после эскалации считается допустимым?
5. Нужно ли хранить в CRM все промежуточные сравнения моделей, которые обсуждал клиент?
6. Какие критерии «не хуже текущего авто» заказчик хочет формализовать (мощность, класс, опции, шумоизоляция и т.п.)?

## 9) Предлагаемые артефакты следующего шага

- BRD-дополнение: «Продажа авто из импорта — conversational flow + pricing transparency».
- SRS-блоки: Calculator API, Trim Knowledge Base, Recommendation Engine.
- BPMN-процесс: Bot qualification → Costing → Recommendation → Manager takeover.
- Data Contract: Lead Profile + Offer Snapshot + Pricing Breakdown.
- UAT-сценарии на базе 5 типовых веток диалога из текущей переписки.
