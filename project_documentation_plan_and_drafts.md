# Комплексный план подготовки проектной документации и консолидированный опросник заказчика

> Статус актуализации: 13.05.2026  
> Основание: консолидированы ответы заказчика, предварительные оценки КП и анализ диалогов из актуальных документов пакета.
> Назначение версии: фиксация входных данных для дальнейшего планирования, диапазонной оценки и подготовки финального КП.

## 1. Основание и цель работы

По итогам предварительного анализа входных материалов (несколько независимых аналитических черновиков по файлам `kurbanov_project_*_pre.md`) зафиксирована единая задача: подготовить управляемый пакет проектных документов для двух продуктовых направлений:

1) Telegram-бот подбора автомобиля с интеллектуальной обработкой запросов, управлением передачей диалога менеджеру и расчетным модулем полной стоимости.
2) Веб-агрегатор логистических предложений с API-матрицей по 15 целевым источникам, первой очередью из 2–4 подключений, унификацией данных и приоритизацией выгодных предложений.

Приоритет поставки зафиксирован отдельно: первым реализуется MVP Telegram-консультанта, а логистический агрегатор оформляется вторым потоком после запуска и стабилизации первого продукта. Второй поток не входит в первый релиз Telegram-консультанта.

Данный документ определяет:
- план работ по выпуску проектных документов (с поэтапной декомпозицией);
- консолидированный опросник заказчика по всем выявленным вопросам;
- черновики ключевых проектных документов в формальном инженерном формате;
- отдельный реестр рисков для внутреннего контурного планирования качества и устойчивости.

## 4. Черновики итоговых проектных документов

## 4.1 Черновик документа «Project Charter / Vision & Scope»

**Наименование инициативы:**
- Продукт А: Интеллектуальный Telegram-бот подбора автомобиля.
- Продукт B: Платформа агрегации логистических предложений.

**Цель:**
Обеспечить управляемую цифровую систему клиентских коммуникаций и обработки рыночных предложений с предсказуемым качеством, прозрачной операционной моделью и масштабируемой архитектурой.

**Границы MVP и второго потока:**
- Продукт А: квалификация обращения, структурированный диалог, доступ к базе знаний, расчетный модуль, управляемая передача менеджеру, базовая аналитика обращений.
- Продукт B: отдельный второй поток после стабилизации MVP Продукта А; первая очередь включает API-матрицу, подключение 2–4 источников, единую витрину объявлений, фильтрацию, историю поиска, избранное, приоритизацию и события уведомлений.

**Ключевые результаты MVP:**
1. Формализованные сценарии работы и критерии приемки.
2. Рабочий контур обработки запросов и данных в продуктивном режиме.
3. Операционные регламенты сопровождения.

## 4.2 Черновик документа «SRS — Требования к системе»

### 1. Функциональные требования (фрагмент)
- FR-A-01: Система должна проводить первичную квалификацию клиента по заданному шаблону параметров.
- FR-A-02: Система должна формировать ответ на основании релевантных материалов базы знаний.
- FR-A-03: Система должна поддерживать режим управляемой передачи диалога ответственному менеджеру.
- FR-A-04: Система должна выполнять расчет итоговой стоимости по утвержденным формулам.
- FR-B-01: Система должна получать данные из утвержденного перечня источников.
- FR-B-02: Система должна приводить входящие данные к единой модели.
- FR-B-03: Система должна предоставлять фильтрацию и сортировку по утвержденным параметрам.
- FR-B-04: Система должна сохранять историю поиска, избранное и параметры сохраненных фильтров.
- FR-B-05: Система должна формировать уведомления по приоритетным предложениям в согласованные каналы.
- FR-B-06: Подключение источников должно выполняться по API-матрице с учетом технических и юридических ограничений площадок.

### 2. Нефункциональные требования (фрагмент)
- NFR-01: Время отклика пользовательского интерфейса в типовом сценарии — в пределах согласованного SLA.
- NFR-02: Все критичные бизнес-события подлежат журналированию.
- NFR-03: Ролевая модель доступа должна быть реализована на уровне API и интерфейса.
- NFR-04: Решение должно поддерживать масштабирование в рамках роста нагрузки.

### 3. Требования к данным (фрагмент)
- DR-01: Для всех сущностей должна быть определена схема обязательных и опциональных полей.
- DR-02: Для ответов, влияющих на коммерческие решения, должна быть доступна ссылка на источник данных.
- DR-03: Для расчетных операций должна храниться версия примененных формул.

## 4.3 Черновик документа «Solution Architecture»

### Логический контур Продукта А
- Канал взаимодействия (Telegram).
- Сервис диалогового управления.
- Сервис генерации ответа и поиска релевантного контекста.
- Хранилище знаний и метаданных.
- Расчетный сервис стоимости.
- Интерфейс менеджера и административные функции.

### Логический контур Продукта B
- Шина интеграций источников.
- Модуль нормализации и контроля качества данных.
- Репозиторий объявлений и индексация.
- Поисковый и фильтрационный слой.
- Модуль событий и оповещений.

### Архитектурные принципы
1. Разделение доменных контуров двух продуктов.
2. Управляемая расширяемость интеграций и источников.
3. Трассируемость решений и наблюдаемость эксплуатационных метрик.
4. Приоритет предсказуемости поведения системы в рабочих сценариях.

## 5. Сравнительная матрица технологических стеков для КП

| Сценарий | Стек | Бюджет | Срок | Отличительные особенности | Ключевые ограничения |
|---|---|---:|---:|---|---|
| Стартовый API | Telegram + backend + коммерческая языковая модель по API + простая база ответов | 350–650 тыс. ₽ | 3–5 недель | Быстрый запуск, низкий входной бюджет, проверка спроса | Нет дообучения, слабее контроль качества, ограниченная база знаний |
| Рабочий MVP | Telegram + backend + локальная база знаний + векторный поиск + калькулятор + админка | 900 тыс.–1,6 млн ₽ | 5–8 недель | Рекомендуемый баланс цены, качества и расширяемости | Нет собственного оборудования, логистика не входит в первый релиз |
| Масштабируемая платформа | MVP + CRM/уведомления + аналитика + контроль качества + резервный поставщик модели + первая очередь логистики | 2,5–5,5 млн ₽ | 3–5 месяцев | Подготовка к долгосрочному развитию и нескольким продуктовым потокам | Требует больше сопровождения и отдельной интеграционной матрицы |
| Флагманский контур | Собственная инфраструктура, GPU-оборудование, команда данных и разработки, цикл обучения и тестирования | 8–18 млн ₽ + 35–65 млн ₽ оборудование | 6–12 месяцев | Максимальный контроль данных и независимость | Оправдан только при высокой нагрузке и постоянной команде |

### Этап 1. Рабочий MVP Telegram-консультанта
- Реализовать управляемый диалог, сбор параметров, расчет стоимости в утвержденном контуре и передачу менеджеру.
- Подключить локальную базу знаний, векторный поиск, журналирование и админку для курсов/контента.
- Провести приемку по сценариям: первичная квалификация, расчет «под ключ», подбор альтернатив, неизвестный вопрос, отключение после подключения менеджера.

### Этап 2. Стабилизация и измерение эффекта
- 1–2 месяца собирать статистику: количество лидов, доля эскалаций, причины отказов, конверсия, экономия времени менеджера.
- Обновить базу знаний по реальным диалогам и зафиксировать правила качества ответов.
- Принять решение о расширении бюджета на платформенный контур.

### Этап 3. Масштабируемая платформа
- Добавить CRM/реестр лидов, расширенную аналитику, роли, уведомления и контроль качества.
- Подготовить второй поток логистического агрегатора: API-матрицу по 15 площадкам, выбор 2–4 источников первой очереди и единую модель данных.
- Реализовать первую очередь агрегатора с фильтрами v1, историей поиска, избранным, уведомлениями и UAT с заказчиком и Тамарой.
- Зафиксировать API-матрицу для дальнейшего подключения остальных площадок после приемки первой очереди.

### Этап 4. Собственная инфраструктура
- Рассматривать только после подтверждения устойчивой нагрузки и экономической целесообразности.
- Подготовить отдельный расчет владения: оборудование, размещение, команда, электричество, резервирование, сопровождение.
- Запускать собственный контур как инвестиционную фазу, а не как условие первого MVP.

## 7. Отдельный реестр рисков (внутренний контур планирования)

> Данный раздел ведется отдельно от клиентских презентационных материалов и используется для внутреннего управления устойчивостью проекта.

### Шаблон реестра
- ID риска
- Описание
- Затрагиваемый контур
- Триггер проявления
- Вероятность / влияние
- Превентивные меры
- План реагирования
- Ответственный
- Статус

### Стартовый перечень категорий
1. Качество и структурированность входных данных.
2. Изменчивость внешних интеграционных источников.
3. Корректность и актуальность расчетной логики.
4. Нагрузочный профиль и изменение паттернов использования.
5. Эксплуатационные и процессные зависимости.

## 8. Рекомендуемый порядок запуска работ
1. Провести интервью с заказчиком по консолидированному опроснику.
2. Зафиксировать ответы в `decision_log.md`.
3. Закрыть формулы калькулятора в `calculator_formula_specification.md`.
4. Заполнить API-матрицу логистики в `logistics_api_matrix.md`.
5. Заполнить реестр корпуса базы знаний в `knowledge_base_corpus_register.md`.
6. Подтвердить коммерческие рамки в `commercial_terms_final_cp_inputs.md`.
7. Утвердить UAT-сценарии в `uat_acceptance_scenarios.md`.
8. Выпустить пакет документов версии v1.0.
9. Провести технический review и нормоконтроль.
10. Обновить точное КП в документе `commercial_proposal_variants_pre.md` или подготовить отдельный финальный документ КП.
11. Обновить итоговый индекс пакета в `project_documentation_package_final.md`.
12. Утвердить baseline и перейти к реализации по волнам.
