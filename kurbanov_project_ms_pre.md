# 🧾 1. Фиксация входных требований (как есть)

## 📌 Основной продукт 1: Telegram-бот по подбору авто

### Функционал:
1. **Диалог с клиентом**
   - Сбор параметров:
     - Марка, модель
     - Цвет салона
     - Пробег
     - Бюджет
   - Уточнение альтернатив (аналогов)

2. **Ответы на вопросы**
   - Бот отвечает автоматически, если знает
   - Если не знает:
     - "Сейчас уточню" / "Подключаю менеджера"
     - Передача диалога менеджеру

3. **База знаний (~5–10 млн символов)**
   - Цены, модели, характеристики
   - Разные страны:
     - Россия
     - Беларусь
     - Казахстан

4. **Калькулятор стоимости**
   - Пользователь вводит цену авто
   - Бот считает итоговую стоимость (пошлины, логистика)
   - Пока только РФ (данные есть только по РФ)

5. **Обучающий контент**
   - Видео-инструкция: как искать авто
   - Ссылки на площадки

6. **Контроль поведения бота**
   - Не отвечать менеджеру
   - Отвечать только клиенту
   - Не генерировать "бред" (ключевой триггер боли)

7. **Нагрузка**
   - Пока ~10 диалогов одновременно

# 🚨 2. Главные риски (ОЧЕНЬ важно для продажи)

## 🔴 Риск 1: "Бот отвечает ерунду"
Причина:
- Отсутствие архитектуры
- Нет контроля контекста
- Нет fallback-логики

✅ Решение:
- RAG (Retrieval-Augmented Generation)
- Контролируемые промпты
- Ограничение зоны знаний

---

## 🔴 Риск 2: База знаний "10 млн символов"
Проблема:
- В лоб не работает (LLM не читает такие объемы)

✅ Решение:
- Векторная база (Pinecone / Weaviate / pgvector)
- Семантический поиск

---

## 🔴 Риск 3: Разные страны и цены
Проблема:
- Динамика цен
- Курсы валют
- Разная логика расчёта

✅ Решение:
- Модуль калькуляций (отдельный сервис)
- Конфигурации по странам

---

## 🔴 Риск 4: Подключение менеджера
Проблема:
- Бот не должен мешать
- Синхронизация диалога

✅ Решение:
- FSM + статус диалога:
  - bot_active
  - human_active

---

## 🔴 Риск 5: Парсинг 10–12 площадок
Проблема:
- API может не быть
- Баны
- Частые изменения верстки

✅ Решение:
- Если API есть → использовать
- Если нет → парсер + прокси + лимиты

---

## 🔴 Риск 6: Клиент скорее всего недооценивает стоимость
Это **ключевой момент**:  
он думает "сделать бота" ≠ реально это:

👉 продукт уровня стартапа

# 🧠 3. Архитектура решения

## 🧩 Общая схема:

```
Telegram Bot
    ↓
Backend (API)
    ├── Dialog Engine (FSM)
    ├── LLM Service (GPT / аналог)
    ├── Knowledge Base (Vector DB)
    ├── Calculator Module
    ├── Admin Panel
    └── Manager Interface
```

---

## 🧪 Технологический стек

### Backend:
- Python (FastAPI) ✅ оптимально
- или Node.js (NestJS)

### AI:
- OpenAI / Azure OpenAI / локальные LLM (если бюджет низкий)
- Embeddings + RAG

### База данных:
- PostgreSQL
- + pgvector (или Pinecone)

### Telegram:
- aiogram / pyTelegramBotAPI

### Парсинг:
- Python (Scrapy / Playwright)

### Frontend (для логистики):
- React / Next.js

### Хостинг:
- AWS / DigitalOcean / Hetzner
