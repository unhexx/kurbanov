# Краткий предварительный опросник и реестр входных данных для этапа подготовки КП

**Версия:** 2.0  
**Статус:** Актуализировано по ответам заказчика  
**Дата актуализации:** 13.05.2026

## 1) Назначение документа
Документ фиксирует подтверждённые входные данные и список недостающей информации для подготовки коммерческого предложения (КП) по двум направлениям:
- **Продукт A:** Telegram-бот подбора автомобиля.
- **Продукт B:** Агрегатор логистических предложений.

---

## 2) Зафиксированные ответы заказчика (консолидировано)

## 2.1 Бизнес-рамка
- Цель по продукту A: разгрузить менеджера на **3–4 часа в день**, выводить более «горячий» лид.
- Цель по продукту B: ускорить и упростить поиск грузов, расширить покрытие площадок помимо ATI.SU.
- KPI: выполнение плана по объёму продаж, конверсия лида в сделку, маржинальность.
- Целевой срок MVP: **1 месяц** (желательно).
- Приоритет: сначала **Telegram-бот (MVP)**.
- Ответственные со стороны заказчика:
  - Telegram-бот: заказчик + Дмитрий.
  - Логистический агрегатор: заказчик + Тамара (главный логист).

## 2.2 Продукт A — Telegram-бот автоподбора
- Диалоговый сценарий подтверждён: сбор 15 параметров клиента + ветвления по бюджету, возрасту авто и стране доставки.
- Правила эскалации в менеджера подтверждены:
  - авто 1–3 года;
  - страны вне РФ/РБ/КЗ;
  - неуверенность бота/нестандартный случай;
  - случаи с электромобилями (приоритет личной коммуникации менеджера).
- Правило отключения/возврата бота:
  - бот отвечает только на входящие;
  - при исходящем сообщении менеджера бот отключается в чате;
  - обратное включение только по команде менеджера (/resume_bot или через админку).
- География и ограничения:
  - калькулятор только для РФ и для авто 3–5 лет до 160 л.с.;
  - в РБ/КЗ калькулятор не применяется, передача менеджеру;
  - праворульные в РБ/КЗ не везём;
  - для Китая выводить предупреждение о правиле 180 дней.
- Источники поиска авто зафиксированы (Япония/Китай/Корея + drom.ru по комплектациям).

## 2.3 Калькулятор (актуализированная фиксация)
- Экспортные: **15 000 ¥** (уточнение заказчика).
- Комиссия: 50 000 ₽.
- Брокер: 75 000–85 000 ₽.
- Доставка до Москвы: ~190 000–200 000 ₽.
- Курсы валют: ввод через админку (в одном ответе подтверждён ручной ввод обеих валют).
- Формула расчёта и диапазоны пошлин/сборов зафиксированы как обязательные для реализации в коде.
- Открытое уточнение: множитель ×1.02 в референсной XLS-таблице.

## 2.4 База знаний и контент
- Форматы: TXT + таблицы + стартовые docx-файлы моделей.
- Владельцы контента: заказчик и Дмитрий.
- Частота обновления: чаще на старте, затем ~раз в 3–4 месяца.
- Жёсткое правило: если в базе нет ответа — бот передаёт менеджеру.

## 2.5 Продукт B — логистический агрегатор
- Зафиксировано 15 целевых бирж-источников (перечень подтверждён заказчиком).
- По API: часть данных и ключей в процессе запроса у площадок; на текущем этапе возможна гибридная стратегия (API + парсинг).
- Критичные фильтры v1:
  - дата погрузки (обязательно),
  - тип кузова,
  - тип загрузки (верх/бок/зад),
  - параметры оплаты,
  - откуда/куда + радиус,
  - вес и объём (диапазоны).
- В выдаче обязательны: маршрут, контакты, наименование груза, габариты/объём, сумма, комментарии, переход в карточку компании.
- Операционный фокус: будние дни, особенно понедельник/вторник.

## 2.6 Интеграции и эксплуатация
- Контур старта: DEV и ограниченная группа логистов.
- Уведомления: желательно Telegram и MAX.
- CRM: желательно, окончательное решение отложено.
- История поиска: обязательное сохранение, избранное, быстрый возврат к фильтру.

## 2.7 Приёмка
- Критерии: функциональная полнота по ТЗ + отсутствие критических дефектов (Severity 1/2).
- UAT: заказчик и Тамара.
- Изменения после согласования КП: через предварительное уведомление заказчиком.

## 3) Реестр входных данных для подготовки КП

## 3.1 Подтверждённые входные данные
1. Цели и KPI по двум продуктам — подтверждены.
2. Приоритет и целевой срок MVP (Telegram-бот, 1 месяц) — подтверждены.
3. Детализированный сценарий и правила handover для бота — подтверждены.
4. Ограничения калькулятора и базовая формула — подтверждены.
5. Список логистических платформ (15) — подтверждён.
6. Ключевые фильтры и состав карточки груза — подтверждены.
7. Ответственные за UAT и владельцы контента — подтверждены.

## 3.2 Недостающие входные данные (блокирует «точное КП»)
1. Детальная API-матрица по 15 биржам (эндпоинты, лимиты, аутентификация, ToS).
2. Финальное решение по множителю ×1.02 и полному алгоритму 1:1 с XLS.
3. Полная ролевая модель и матрица прав (бот/админка/агрегатор).
4. Полный пакет базы знаний FAQ (целевые 5–10 млн символов) в рабочей структуре.
5. Формальные UAT-сценарии по каждому критичному бизнес-кейсу.
6. Подтверждённый формат CRM-интеграции и событий уведомлений.
7. Согласованные SLA обновления данных в агрегаторе (внутридневные/ежедневные/иные).
8. Коммерческие рамки: бюджетный диапазон и целевая модель этапной оплаты (50/50 или 70/30).

## 4) Предварительные коммерческие сценарии

## 4.1 Принятые допущения для оценки
- Основной фокус следующего КП: **Telegram-бот / фирменный консультант по подбору авто**.
- Логистический агрегатор оценивается как отдельная фаза развития после запуска первого продукта.
- Формат оценки: диапазоны бюджета и сроков, без фиксации договорной цены до закрытия открытых вопросов.
- Позиционирование консультанта: клиент видит фирменный сервис компании; внутренний технологический стек не раскрывается, но не утверждается, что отвечает конкретный сотрудник.
- Базовая модель оплаты для MVP: 50/50; 70/30 можно использовать при срочном старте и согласованном составе работ.

## 4.2 Вилки бюджета для приценки
| Сценарий | Ориентир бюджета | Срок | Назначение |
|---|---:|---:|---|
| Стартовый API-вариант | 350–650 тыс. ₽ | 3–5 недель | Быстрая проверка гипотезы и базовая квалификация лидов |
| Рабочий MVP | 900 тыс.–1,6 млн ₽ | 5–8 недель | Рекомендуемый стартовый продукт для эксплуатации |
| Масштабируемая платформа | 2,5–5,5 млн ₽ | 3–5 месяцев | Развитие аналитики, CRM, качества и первой очереди логистики |
| Флагманский контур | 8–18 млн ₽ разработка + 35–65 млн ₽ оборудование | 6–12 месяцев | Собственная инфраструктура и команда развития |

## 4.3 Ограничения бюджетных вариантов
- Бюджет уровня **26 тыс. ₽** может покрыть только демонстрационный скрипт: несколько заготовленных ответов и простую связку с Telegram. Для рабочего продукта этого недостаточно, так как не покрываются база знаний, калькулятор, handover, тестирование, журналирование и сопровождение.
- Стартовый API-вариант не дает полноценного контроля качества, не включает дообучение, развитую локальную векторную базу и промышленную админку контента.
- Рабочий MVP не включает собственное оборудование, но закладывает архитектуру для дальнейшего переноса части контуров на контролируемую инфраструктуру.
- Масштабируемая платформа требует отдельного бюджета на интеграции и сопровождение.
- Флагманский контур экономически оправдан только после подтверждения устойчивой нагрузки, достаточного массива данных и готовности содержать команду развития.

## 4.4 Рекомендуемая стратегия
Рекомендуется начинать с **рабочего MVP**: это минимальный вариант, который можно использовать в продажах и развивать без полной переделки. Стартовый API-вариант дешевле, но больше похож на проверку идеи. Флагманский контур с собственным оборудованием целесообразно рассматривать как инвестиционную фазу после 1–2 месяцев эксплуатации MVP и подтверждения бизнес-эффекта.

---

## 5) Критерий готовности к выпуску точного КП
Точное КП выпускается после закрытия пунктов раздела 3.2 и письменного подтверждения раздела 3.1 ответственными лицами заказчика.

До этого момента допускается только диапазонная оценка с декомпозицией по волнам (MVP → v1 → масштабирование).

Связанный документ с вариантами оценки: `commercial_proposal_variants_pre.md`.
